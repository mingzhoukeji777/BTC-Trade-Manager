BTC 长线主仓位管理系统确认版思维导图

确认后的核心：Binance U本位永续 和 OKX 币本位永续 是两个独立长线主仓系统，可通过切换按钮/下拉框选择；每个主仓下面挂加仓、对冲、减仓、止盈、止损等节点，每个节点都像“小主仓位分支”一样独立计算盈亏、持仓、手续费、成本止损和移动止盈。

橙色 = 主仓核心绿色 = 节点/保护规则红色 = 风险/异常/待处理灰色 = 数据/API/记录

确认后的核心软件模块

  • 账户选择 Binance U本位主仓 / OKX 币本位主仓，独立切换，互不混淆。
  • 主仓树 M 主仓下面挂 A 加仓、H 对冲、TP 止盈、SL 止损、R 减仓。
  • 小主仓分支 每个加仓/对冲都独立显示持仓大小、浮盈亏、手续费、盈亏平衡、成本止损、移动止盈。
  • 待归类订单 API 获取订单后无法自动判断关系时，进入待处理池，手动绑定节点类型和父节点。
  • 情景模拟 输入 BTC 目标价，模拟主仓和所有分支的盈亏、强平距离、建议动作。

编号规则建议

节点类型编号说明
主仓M1 / M2一个牛熊周期的核心仓位
加仓A1 / A2挂在某个主仓下，可有自己的止盈止损
对冲H1 / H2反向保护仓，需绑定开/平订单
止盈TP1 / TP2分批止盈或移动止盈
止损SL1 / SL2成本止损或保护止损
减仓R1 / R2降低主仓风险或释放保证金

费用与真实成交价规则

  • 默认手续费：统一按 0.05% 单向 估算。
  • 真实手续费：如果 API 能获取订单真实手续费，则用真实手续费覆盖默认估算，并计入主仓/加仓/对冲/止盈节点。
  • 滑点：不做估算，直接使用 API 获取到的真实成交开仓价和平仓价。
  • 资金费率：通过 API 获取每次结算资金费，归入对应交易所主仓位成本/收益，并影响总盈亏。

订单自动匹配逻辑

情况处理方式
开仓订单能通过时间/方向/数量/价格匹配到某个计划节点自动绑定到 A/H/M 节点，并记录交易所订单号。
平仓订单与开仓订单号不同通过成交方向、数量、时间、价格区间、已有节点状态进行候选匹配。
无法确定是加仓、对冲、止盈还是止损进入“待归类订单池”，弹出提示，手动选择节点类型和父节点。
部分成交/多笔成交合并为一个节点成交明细，也保留每笔原始订单记录。

下一步重构管理软件 v0.2 的准确目标

  • 删除“短线/波段/长线账户”框架，改为 Binance 主仓 / OKX 主仓 独立切换。
  • 建立数据库表：主仓表、节点表、订单映射表、手续费/资金费表、情景模拟表。
  • 主界面做成：左侧主仓树，中间风险总控，右侧节点明细，底部风险提示。
  • 每个加仓/对冲节点都作为独立分支计算，不再只是普通交易记录。
  • 先不自动交易，只做记录、计算、提示和推荐动作。
确认结论：这个确认版结构才是后续 C# 管理软件应采用的主框架。之前 v0.1 只保留少量读取采集器和 SQLite 的基础代码，业务模型要重构。
